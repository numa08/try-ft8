# SPEC: ブラウザ FT8 名刺交換デモ(WSJT-X 簡易版)

- 状態: Draft(フェーズ3レビュー反映済み。残 OQ-4 のみ)
- 作成日: 2026-07-01
- 対象: ブラウザ上で動作する FT8 メッセージング・デモ(Cloudflare Workers/Pages にデプロイ)

## 1. 目的 / 背景

LT で FT8 の仕組みを説明するために、参加者が自分のスマホで実際に FT8 音声を送受信し、3 文字のハンドルネームを交換して回るデモを行う。交換結果はログに残り、その場にいる人同士の自己紹介・名刺交換として機能する。見せ場は「音声だけで名前が交換され、ログに積み上がっていく」体験そのもの。参加者は最大 7 名程度、約 10 分の時間制限で終了する。

**成功条件**: 参加者のおよそ半数以上が、最低 1 回の交信成立を体験できること。これはデモ全体の結果指標であり、自動テストではなく当日の運用で判断する。

## 2. ユーザーとシナリオ

- 対象ユーザー: LT 参加者(最大 7 名)。スマホのブラウザでページを開いて操作する。
- ハッピーパス:
  1. ページを開くと名前入力ダイアログ(注意文つき)が出る。大文字英数字ちょうど 3 文字で名前を入力。
  2. 「開始」ボタンを押すと AudioContext が確立され、マイク利用許可を要求。承認すると受信モードになる。
  3. 誰かが `Start Sequence`(CQ 送出)を押すと、その端末が CQ を音声送信する。
  4. CQ を受信した端末が応答し、交互 QSO(`相手 DE 自分` → 双方 `73`)でシーケンスが自動進行する。
  5. シーケンスが完了すると相手の名前がログ(LocalStorage)に保存され、交信人数が更新される。
  6. これを全員が繰り返し、時間制限で終了。

## 3. メッセージ形式(設計の要)

FT8 のフリーテキストは 13 文字上限・大文字英数字とスペースのみ。この器に収めるための制約:

- 名前は **大文字英数字(`A-Z`, `0-9`)ちょうど 3 文字**。
- メッセージ書式は `<相手> DE <自分>`(CQ は `CQ DE <自分>`)、完了時は末尾に ` 73`。
- 最長メッセージは完了時の `XXX DE YYY 73` = **13 文字ちょうど**(名前 3+3、` DE ` 4、` 73` 3)。これが上限に張り付くため、名前を 4 文字にすると溢れる。ゆえに 3 文字固定は必須制約。
- 交互 QSO の並び(A=CQ 発信・EVEN、B=応答・ODD):
  1. A → `CQ DE A`(EVEN)
  2. B → `A DE B`(ODD)
  3. A → `B DE A 73`(EVEN)
  4. B → `A DE B 73`(ODD)
  5. 双方が 73 を送出 → 完了

## 4. 受入基準(Acceptance Criteria)

> すべて EARS 記法。1 項目 1 検証点。詳細パターンは references/ears-notation.md 準拠。

### オンボーディング

- `AC-1` いつ ページを開いたとき、システムは 名前入力ダイアログを表示し、その中に「マイクで音声を収録する」「スピーカーから音が出る」「収録音声は保存・送信しない」旨の注意文を表示しなければならない。
- `AC-2` もし 入力された名前が大文字英数字(`A-Z`, `0-9`)ちょうど 3 文字でなければ、システムは 確定を拒否し理由を表示しなければならない。
- `AC-3` いつ 名前が確定され「開始」ボタンが押されたとき、システムは AudioContext を確立し、マイク利用許可を要求しなければならない。
- `AC-4` いつ マイク許可が承認されたとき、システムは 受信モードのメイン画面に遷移しなければならない。
- `AC-5` もし マイク許可が拒否されたら、システムは 送受信できない旨を表示しなければならない(再試行はページ再読み込みで行う)。
- `AC-6` いつ ユーザーが運用中に名前を変更したとき、システムは 3 文字ルールで検証し、以降の送信で新しい名前を用いなければならない。

### 時刻同期・送受信スロット

- `AC-7` システムは 端末時計に基づき 15 秒スロット(:00 / :15 / :30 / :45)に同期し、スロット境界から ±500ms 以内に送信を開始しなければならない。
- `AC-8` CQ を送出する局として振る舞う間、システムは EVEN スロット(:00 / :30)で送信しなければならない。
- `AC-9` 他局の CQ に応答する局として振る舞う間、システムは ODD スロット(:15 / :45)で送信しなければならない。
- `AC-10` システムは 起動時にランダムな送信周波数オフセットを選択しなければならない。
- `AC-30` システムは 画面に time.is への外部リンクを表示し、デコード不良時に時刻ずれを確認できるようにしなければならない。

### 受信・デコード

- `AC-11` 受信モードの間、いつ FT8 音声のデコードに成功したとき、システムは デコード結果をテキストとして表示しなければならない。

### メッセージ形式(不変条件)

- `AC-25` システムが送信するすべての FT8 メッセージ(CQ・応答・73 を含む)は、13 文字を超えてはならない。

### CQ 送出(自局発信)

- `AC-12` いつ `Start Sequence` ボタンが押されたとき、システムは 次の EVEN スロットで `CQ DE <自分>` をエンコードして再生しなければならない。
- `AC-13` いつ CQ を送出したとき、システムは UI を送信中/応答待ち状態に更新しなければならない。
- `AC-14` もし 自局の CQ に対する応答が得られないまま次の送信機会が来たら、システムは 最大 3 回まで CQ を再送しなければならない。
- `AC-15` もし 3 回の再送後も応答が無ければ、システムは 待機状態に遷移しなければならない。

### CQ への応答(他局への着信)

- `AC-16` いつ 他局の CQ をデコードしたとき、システムは その CQ に対する応答シーケンスを開始しなければならない。
- `AC-17` もし デコードした CQ の送信元が既に LocalStorage のログに存在するなら、システムは その CQ を無視しなければならない。
- `AC-18` いつ 自局の CQ に対して複数の応答をデコードしたとき、システムは サブバンド周波数が最も低い応答局にのみ応答しなければならない。
- `AC-19` もし 応答シーケンス中、相手からの返信が無いまま次の送信機会が来たら、システムは 最大 3 回まで直前のメッセージを再送しなければならない。
- `AC-20` もし 3 回の再送後も返信が無ければ、システムは 別の CQ をデコードするまで待機しなければならない。

### 交互 QSO シーケンス

- `AC-26` いつ 応答対象の CQ を選択したとき(応答局として)、システムは 次の ODD スロットで `<相手> DE <自分>` を送信しなければならない。
- `AC-27` いつ 自局の CQ への応答 `<自分> DE <相手>` をデコードしたとき、システムは 次の EVEN スロットで `<相手> DE <自分> 73` を送信しなければならない。
- `AC-28` いつ 相手からの `<自分> DE <相手> 73` をデコードしたとき、システムは 次の ODD スロットで `<相手> DE <自分> 73` を送信しなければならない。
- `AC-29` いつ 自局と相手の双方が 73 を送信し終えたとき、システムは シーケンスを完了とみなさなければならない。

### シーケンス完了・ログ

- `AC-21` いつ 交信シーケンスが完了したとき、システムは 交信相手の名前を含むログを LocalStorage に保存しなければならない。
- `AC-22` システムは 保存済みログを一覧表示する UI を提供しなければならない。
- `AC-23` いつ ユーザーがログ削除を実行したとき、システムは 対象ログを LocalStorage から削除しなければならない。
- `AC-31` システムは これまでに交信が成立した相手の人数(N 人)を表示しなければならない。

### プライバシー

- `AC-32` システムは マイクで取得した音声を、デコード以外の目的で保存・送信してはならない。

### 空き箱防止(起動・描画確認)

- `AC-24` システムを起動したとき、名前入力ダイアログ(注意文つき)が表示され、名前確定と「開始」・マイク許可承認を経て、`Start Sequence` ボタンとログ一覧を含む受信モードのメイン画面が描画されること(Playwright で確認)。

## 5. 制約

- 技術/環境: ブラウザ(モバイル Safari / Chrome、iOS・Android)。FT8 のエンコード/デコードは WASM(ft8_lib 等、フリーテキストモード)。音声入出力は Web Audio API。永続化は LocalStorage。
- AudioContext: 自動応答(73 等の自動送信)のため、初回の「開始」ボタン(ユーザージェスチャ)で AudioContext を確立し、以後維持する。
- 時刻: 端末時計をそのまま使用(補正なし)。ズレ耐性は ±500ms。ユーザーは time.is リンクで確認できる。
- デプロイ: Cloudflare Workers または Pages(getUserMedia のため HTTPS 必須)。
- 既存コードとの境界: 新規プロジェクト。RF ハードウェアや外部サービス連携はなし。

## 6. テスト方針

- `AC-11` のデコード検証は、既知の FT8 音声フィクスチャ(事前生成した .wav)を用意し、(a) エンコード↔デコードの往復テストと (b) ゴールデン音声のデコードテストで確認する。
- `AC-25`(13 文字以内)はメッセージ生成の純関数テストで確認する。
- `AC-7`(±500ms)はスロット境界からの送信開始時刻を計測して確認する。
- `AC-24` は Playwright による起動〜メイン画面描画の e2e で確認する。

## 7. 非目標(Non-Goals)

- 本物の FT8 信号のデコード / FT8CN・WSJT-X 等の実アプリとの相互運用。
- 本物のコールサイン(6 文字)対応 / FT8 標準メッセージ形式(コールサインの構造的エンコード)。フリーテキスト方式に統一する。
- 名前(3 文字)衝突の自動回避。被った場合はその場で手動修正する運用。
- アカウント連携、ログのクラウド保存・集約。
- SNR(信号レポート)算出、グリッドロケータ交換、バンド切り替え。
- ODD/EVEN の手動選択 UI(役割で自動決定するため不要)。
- サブバンド衝突時の自動回避(受信不良時は送信側の手動リロードでオフセットを引き直す運用でカバー)。
- 応答先の手動選択(サブバンド周波数順で自動)。
- 全体進捗・残り人数の可視化(その場の人数が不定のため不要。自局の交信人数 N は AC-31 で表示)。
